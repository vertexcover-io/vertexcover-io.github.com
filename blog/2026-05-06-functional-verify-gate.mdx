<TLDR>

When an agent writes the code, a passing test suite no longer tells you the feature works. The agent will happily write tests that pass against a feature that's broken. So we added a second gate that runs after the unit suite: it boots the app, hits the live API, queries the live database, and drives the UI in a real browser, then writes a proof report the reviewer can replay. Most "quality gates" answer one question: did the build go green? If type checker, lint, and tests are all clean, ship it.

The problem is, the agent ships a bug every week that fits cleanly through that question. The code compiles. The tests it wrote pass. The endpoint still returns the wrong shape, the row never makes it into the database, or the page renders broken, because nothing in the pipeline ever booted the application and exercised it.

So we added a second check that does. It is called `functional-verify`. It runs after the metric check, as a separate sub-agent, and its job is to run the live app: API endpoints via `curl`, database state via real queries, UI via Playwright MCP. We have been using it on every PR for about a month, and at this point we lean on it more than we lean on the unit suite.

## What the metric check already covers

Before `functional-verify` runs, `quality-gate` has already enforced nine checks against a baseline captured at branch start. Three of them catch failure modes a normal CI pipeline does not:

<ol style={{ listStyleType: 'decimal', listStylePosition: 'outside', paddingLeft: '1.5rem', margin: '12px 0' }}>
  <li><strong>Test count must be greater than or equal to baseline.</strong> Catches the agent silently deleting a failing test instead of fixing it. We have watched this happen.</li>
  <li><strong>Scope compliance.</strong> <code>git diff --name-only</code> is intersected with the plan's declared file list. Anything outside that intersection is flagged. Stops the agent from "helpfully" refactoring 14 unrelated files.</li>
  <li><strong>Ignore comment audit.</strong> <code>git diff</code> is grep'd for <code>@ts-ignore</code>, <code># noqa</code>, <code>eslint-disable</code>, and friends. Any new ignore without an inline justification is a hard fail.</li>
</ol>

The metric check is good at this. It is bad at "the endpoint returns 200 but the row was never inserted", "the listing markdown contains relative URLs the LLM will never match", and "the page renders, but the headline anchor is 24 pixels tall and a thumb can't tap it." Those are what `functional-verify` is for.

## What functional-verify actually does

The skill reads a `## Verification Scenarios` section from the spec, or derives one scenario per acceptance criterion if the section is absent. Then it runs.

<ol style={{ listStyleType: 'decimal', listStylePosition: 'outside', paddingLeft: '1.5rem', margin: '12px 0' }}>
  <li><strong>Start the app.</strong> If it's already running locally, attach to it. Otherwise, boot it from the project's standard start scripts and wait until it responds.</li>
  <li><strong>Exercise the API.</strong> For each scenario in the spec, send the request the spec describes and record exactly what came back: the command sent, the status code, and the response body.</li>
  <li><strong>Exercise the database.</strong> For each scenario, run the validation query against the live database and compare what's actually in there against what the spec says should be.</li>
  <li><strong>Exercise the UI.</strong> For each scenario, drive a real browser through the page: navigate, click, type, fill forms, wait for things to appear. Take screenshots at phone, tablet, and desktop widths. For long pages, take a series of overlapping screenshots so nothing is missed below the fold. Capture browser console errors after every page load.</li>
  <li><strong>Write the proof report.</strong> A verdict for each scenario, plus a coverage table that maps every requirement and edge case in the spec to the evidence file that proves it.</li>
  <li><strong>Clean up.</strong> If the run started the app in step 1, shut it down.</li>
</ol>

The interesting parts are how it grades what comes back from each of those layers, and what counts as evidence.

## Every claim cites evidence the reviewer can replay

Hand a vision model a screenshot and ask if the page looks fine, and it will tell you yes. Hand it a 200 response and ask if the API works, same answer. The defense has to be structural, so the skill rejects bare adjectives. Every claim must be grounded in something the reviewer can rerun: an HTTP response body, a SQL row, a `getBoundingClientRect()` rect, a quoted DOM string, a computed style value. "Polished", "looks off", or "works fine" are explicitly banned.

### A backend regression caught mid-run

A change landed that passed every unit test and would have silently shipped an empty newsletter the next morning.

The agent had just swapped out the component that fetches a page and converts it to markdown. Downstream, an LLM reads that markdown, picks the URLs worth indexing, and returns them as a list. A matcher then checks each returned URL against the markdown with a literal substring search. After the swap, the converter started emitting **relative** paths (`/news/claude-opus-4-7`) while the LLM kept returning **absolute** URLs (`https://anthropic.com/news/claude-opus-4-7`). The substring check never matched. Every chosen post would have been silently dropped.

The reason the unit tests missed it is the part worth remembering: every test fixture in the suite happened to use absolute URLs, so the substring check always matched in tests. The verifier was the first thing in the pipeline to ever hand the converter a real production page. The bug had been reachable for as long as the suite existed; nothing had ever reached it.

The fix was a one-liner that resolves `<a href>` against the page's base URL before the converter sees the HTML.

One thing worth calling out about how the verifier reported its run. A separate scenario in the same report failed for an unrelated reason: an external API key was out of credit. The verifier didn't paper over it or claim a pass. It logged the error, marked the scenario as an external blocker, and noted that the same code path would have failed identically before the change, so it wasn't a regression. That honesty about *what was and wasn't verified* is the part we care about most.

### A UI regression the test suite was never going to catch

Different feature, same skill. The agent had just made the public archive page mobile-friendly. Tests passed. Opened it on a laptop, looked great.

The verifier opened the same page on a phone-sized window and tried to tap a story. It couldn't. The headlines were rendered as thin strips of text, much shorter than a thumb is wide, so any attempted tap landed somewhere in a cluster of three headlines and never reliably on the one you wanted. The agent had built a list you could not actually use on a phone.

The verifier put the actual numbers in the report: each headline was 24 pixels tall. A comfortable tap target on a phone is at least 44. That's the whole bug, in one sentence the reviewer can re-check.

The reason the test suite missed it is the reason it always misses bugs of this shape. Unit tests render components in isolation, on a desktop, with a mouse. They don't know what a phone screen feels like. A real browser, opened to phone size, does, and the check runs that browser on every PR.

## Why the verifier can't know the implementer

`functional-verify` runs as a separate sub-agent with no view of the code that produced the feature. The dev agent wrote the code and the tests; the verifier only sees a running application and a spec. No shared memory between them, and no shared incentive to declare the work done.

That separation is the whole point. In a healthy engineering team, the person who tests a feature is structurally allowed to say no to the person who built it. Without a second human in the loop, the pipeline has to simulate that separation itself. If the same agent that writes the feature also writes the "verified, all green" report, the report will say green every single time.

So if you manage engineers who use coding agents, the first question worth asking is who in the loop is structurally allowed to say no. If the answer is the agent that wrote the code, the check isn't really a check.

## How the proof report ends up on the PR

Every run of `functional-verify` writes one folder:

```
docs/spec/<feature>/verification/proof-report.md
docs/spec/<feature>/verification/api/<scenario-id>.txt
docs/spec/<feature>/verification/static/<scenario-id>.txt
docs/spec/<feature>/verification/ui/<route>-<viewport>.png
docs/spec/<feature>/verification/ui/<route>-<viewport>-slice-NN.png
docs/spec/<feature>/verification/ui/observations.md
```

The orchestrator's commit stage picks all of it up and includes it in the PR. The reviewer opens the PR, opens `proof-report.md`, and walks through:

<ol style={{ listStyleType: 'decimal', listStylePosition: 'outside', paddingLeft: '1.5rem', margin: '12px 0' }}>
  <li>A summary table: scenario id, type (api / db / ui / static), description, verdict.</li>
  <li>API evidence: curl commands and response bodies.</li>
  <li>DB evidence: queries and the rows they returned.</li>
  <li>UI evidence: one row per (route, viewport) pointing at the saved PNG.</li>
  <li>Spec coverage: every requirement and edge case from the spec, mapped to the evidence file that proves it. A row without evidence is a gap, listed as NOT VERIFIED with a reason.</li>
  <li>A "Not executed" section listing things the skill could not check (touch-hold gestures, real-device sensors, no-new-deps assertions, external services that were unreachable). Listed honestly, not papered over.</li>
</ol>

If the reviewer disagrees with a verdict, they have the rect, the curl, the SQL row, the file path, or the screenshot to argue against.

Today the proof report lands on the PR for human review. The reviewer reads the verdicts, opens the screenshots, and decides whether to approve. The check produces evidence; the human still decides. That distinction matters: the report is not an automatic merge blocker, but it is the artifact we open first before clicking approve.

A month into running this on every PR, the thing that's changed isn't the bug count. It's where we look for bugs. The unit suite still runs, still goes green, and we still glance at it. But the artifact we actually open before approving a PR is the proof report. That's the file that tells us the feature works for the person who's going to use it, not just the person who wrote it.
