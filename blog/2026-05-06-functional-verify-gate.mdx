Most "quality gates" answer one question: did the build go green? If type checker, lint, and tests are all clean, ship it.

The problem is, the agent ships a bug every week that fits cleanly through that question. The code compiles. The tests it wrote pass. The endpoint still returns the wrong shape, the row never makes it into the database, or the page renders broken, because nothing in the pipeline ever booted the application and exercised it.

So we added a second gate that does. It is called `functional-verify`. It runs after the metric gate, as a separate sub-agent, and its job is to run the live app: API endpoints via `curl`, database state via real queries, UI via Playwright MCP. We have been using it on every PR for about a month, and at this point we lean on it more than we lean on the unit suite.

## What the metric gate already covers

Before `functional-verify` runs, `quality-gate` has already enforced nine checks against a baseline captured at branch start. Three of them catch failure modes a normal CI pipeline does not:

<ol style={{ listStyleType: 'decimal', listStylePosition: 'outside', paddingLeft: '1.5rem', margin: '12px 0' }}>
  <li><strong>Test count must be greater than or equal to baseline.</strong> Catches the agent silently deleting a failing test instead of fixing it. We have watched this happen.</li>
  <li><strong>Scope compliance.</strong> <code>git diff --name-only</code> is intersected with the plan's declared file list. Anything outside that intersection is flagged. Stops the agent from "helpfully" refactoring 14 unrelated files.</li>
  <li><strong>Ignore comment audit.</strong> <code>git diff</code> is grep'd for <code>@ts-ignore</code>, <code># noqa</code>, <code>eslint-disable</code>, and friends. Any new ignore without an inline justification is a hard fail.</li>
</ol>

The metric gate is good at this. It is bad at "the endpoint returns 200 but the row was never inserted", "the listing markdown contains relative URLs the LLM will never match", and "the page renders, but the headline anchor is 24 pixels tall and a thumb can't tap it." Those are what `functional-verify` is for.

## What functional-verify actually does

The skill reads a `## Verification Scenarios` section from the spec, or derives one scenario per acceptance criterion if the section is absent. Then it runs.

```
1. Probe common ports (3000, 5173, 8080, ...).
   If nothing answers, start the app from package.json scripts
   or compose.yml. Poll for health up to 30s.

2. For each API scenario:
     curl the endpoint with the spec's payload.
     Capture exact command, status, response (<= 50 lines).

3. For each DB scenario:
     run the validation query against the live database.
     Compare actual to expected.

4. For each UI scenario, drive a real Chrome via Playwright MCP (Model Context Protocol):
     browser_navigate, browser_click, browser_type,
     browser_fill_form, browser_wait_for.
     Screenshot at viewports 375 / 768 / 1280 per route.
     Capture viewport-scale slices when scrollHeight > 2 * vh.
     browser_console_messages after every navigation.

5. Write proof-report.md with verdict per scenario,
   plus a spec-coverage table mapping each REQ-N / EDGE-N
   to the evidence file that proves it.

6. Kill any process started in step 1.
```

The interesting parts are how it grades what comes back from each of those layers, and what counts as evidence.

## Every claim cites evidence the reviewer can replay

Hand a vision model a screenshot and ask if the page looks fine, and it will tell you yes. Hand it a 200 response and ask if the API works, same answer. The defense has to be structural, so the skill rejects bare adjectives. Every claim must be grounded in something the reviewer can rerun: an HTTP response body, a SQL row, a `getBoundingClientRect()` rect, a quoted DOM string, a computed style value. "Polished", "looks off", or "works fine" are explicitly banned.

### A backend regression caught mid-run

The newsletter pipeline fetches articles, converts each page to markdown, and hands the markdown to an LLM that picks out the post URLs worth indexing. The agent had just swapped the old fetcher for one built on [Crawlee](https://crawlee.dev), a web-scraping library. Unit tests passed.

`functional-verify` booted the stack and triggered a real run via `POST /api/runs/now`. The new converter was emitting **relative** paths (`/news/claude-opus-4-7`). The LLM downstream returns **absolute** URLs. The matcher between them does a literal substring check, so every chosen post would have been silently rejected and tomorrow's newsletter would have arrived empty.

The fix was a one-liner in `convert.ts` that resolves `<a href>` attributes against the page's baseUrl before the markdown converter sees the HTML. The test fixtures all happened to use absolute URLs already, so the substring check always matched in unit tests. The verifier was the first piece of the pipeline to ever hand the converter a real production page.

A sibling entry in the same report, `VS-8`, is also worth noting. The LLM call further down the pipeline failed because the project's API key was out of credit. The verifier didn't paper over it: it logged the error, marked the scenario `EXTERNAL BLOCKER`, and noted the failure was not a regression because the same code path would fail identically under the previous implementation. That honesty about what was and wasn't verified is the part we care about most.

### A UI regression the test suite was never going to catch

Different feature, same skill. The agent had just made the public archive page mobile-friendly. Tests passed. Opened it on a laptop, looked great.

The gate opened the same page on a phone-sized window and tried to tap a story. It couldn't. The headlines were rendered as thin strips of text, much shorter than a thumb is wide, so any attempted tap landed somewhere in a cluster of three headlines and never reliably on the one you wanted. The agent had built a list you could not actually use on a phone.

The verifier put the actual numbers in the report: each headline was 24 pixels tall. A comfortable tap target on a phone is at least 44. That's the whole bug, in one sentence the reviewer can re-check.

The reason the test suite missed it is the reason it always misses bugs of this shape. Unit tests render components in isolation, on a desktop, with a mouse. They don't know what a phone screen feels like. A real browser, opened to phone size, does, and the gate runs that browser on every PR.

## Why the verifier can't know the implementer

`functional-verify` runs as a separate sub-agent with no view of the code that produced the feature. The dev agent wrote the code and the tests; the verifier only sees a running application and a spec. No shared memory between them, and no shared incentive to declare the work done.

That separation is the whole point. In a healthy engineering team, the person who tests a feature is structurally allowed to say no to the person who built it. Without a second human in the loop, the pipeline has to simulate that separation itself. If the same agent that writes the feature also writes the "verified, all green" report, the report will say green every single time.

So if you manage engineers who use coding agents, the first question worth asking is who in the loop is structurally allowed to say no. If the answer is the agent that wrote the code, the gate isn't really a gate.

## How the proof report ends up on the PR

Every run of `functional-verify` writes one folder:

```
docs/spec/<feature>/verification/proof-report.md
docs/spec/<feature>/verification/api/<scenario-id>.txt
docs/spec/<feature>/verification/static/<scenario-id>.txt
docs/spec/<feature>/verification/ui/<route>-<viewport>.png
docs/spec/<feature>/verification/ui/<route>-<viewport>-slice-NN.png
docs/spec/<feature>/verification/ui/observations.md
```

The orchestrator's commit stage picks all of it up and includes it in the PR. The reviewer opens the PR, opens `proof-report.md`, and walks through:

- A summary table: scenario id, type (api / db / ui / static), description, verdict.
- API evidence: curl commands and response bodies.
- DB evidence: queries and the rows they returned.
- UI evidence: one row per (route, viewport) pointing at the saved PNG.
- Spec coverage: every REQ-N and EDGE-N from the spec, mapped to the evidence file that proves it. A row without evidence is a gap, listed as NOT VERIFIED with a reason.
- A "Not executed" section listing things the skill could not check (touch-hold gestures, real-device sensors, no-new-deps assertions, external services that were unreachable). Listed honestly, not papered over.

If the reviewer disagrees with a verdict, they have the rect, the curl, the SQL row, the file path, or the screenshot to argue against.

Today the proof report lands on the PR for human review. The reviewer reads the verdicts, opens the screenshots, and decides whether to approve. The gate produces evidence; the human still decides. That distinction matters: the report is not an automatic merge blocker, but it is the artifact we open first before clicking approve.

A month into running this on every PR, the thing that's changed isn't the bug count. It's where we look for bugs. The unit suite still runs, still goes green, and we still glance at it. But the artifact we actually open before approving a PR is the proof report. That's the file that tells us the feature works for the person who's going to use it, not just the person who wrote it.
